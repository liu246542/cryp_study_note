    OXT协议

    Info

    Highly-scalable searchable symmetric encryption with support for Boolean queries

    CRYPTO 2013 (CCF A 类会议)

    David Cash, Stanislaw Jarecki, Charanjit Jutla, Hugo Krawczyk, Marcel-Catalin Rosu, Michael Steiner

    "This work presents the design and analysis of the first searchable symmetric encryption (SSE) protocol that supports conjunctive search and general Boolean queries ..."

    1. 系统模型

    数据库由 d 个文件组成，每一个都包含一组关键词 \mathsf{W}_i。SSE 协议输出的是文件标识 \mathsf{ind}，客户端可以通过这些标识来获取加密的文件，之后再进行解密。

    数据库 \mathsf{DB} = (\mathsf{ind}_i, \mathsf{W}_i)_{i=1}^d，是一组由标识符/关键词集组成，其中 \mathsf{ind}_i\in\{0,1\}^\lambda，\mathsf{W}_i \subseteq\{0,1\}^*。那么所有的关键词集合可以表示为 \mathsf{W} = \cup_{i=1}^d \mathsf{W}_i。 由于 \mathsf{ind} 是混淆后的文件标识，因此可以泄露给外包服务器。 对于关键词 \overline{w}\in\mathsf{W}^*，\psi(\overline{w}) 表示对关键词 \overline{w} 的布尔表达式。 我们用 \mathsf{DB}(\psi(\overline{w})) 来表示满足 \psi(\overline{w}) 的表达符。d 表示文件数量，m=|\mathsf{W}|，N=\sum_{w\in \mathsf{W}}|\mathsf{DB}(w)|。

    定义 tuple set，或者称为 T-set。T-set 允许在数据库里将每一个关键词都关联一个固定大小的数据元组。用户查询时，返回的就是这些数据元组列表。在本文提出的协议中，T-set 被用来作为 expanded inverted index。

    T-set 由三个函数组成，\Sigma = (\mathsf{TSetSetup}, \mathsf{TSetGetTag}, \mathsf{TSetRetrieve})：

    • \mathsf{TSetSetup}(\mathbf{T}) \to (\mathsf{TSet}, K_T)
    • \mathsf{TSetGetTag}(K_T,w)\to \mathsf{stag}
    • \mathsf{TSetRetrieve}(\mathsf{TSet}, \mathsf{stag}) \to \mathbf{T}[w]

    2. 单关键词可搜索加密 （SKS）

    SKS，single-keyword SSE Scheme 由以下几个算法组成：

    \mathsf{EDBSetup}(\mathsf{DB}):

    • 选择 PRF F 的密钥 K_S，将 \mathsf{DB} 解析成 (\mathsf{ind}_i, \mathsf{W}_i)_{i=1}^d 的形式
    • 初始化 \mathbf{T} 为一个空数组，关键词集合 \mathsf{W}=\cup_{i=1}^d \mathsf{W}_i
    • 对于每一个 w\in\mathsf{W}，按照以下方式构建 \mathbf{T}[w]：
      • 初始化一个空列表 \mathbf{t}，令 K_e\gets F(K_S, w)
      • 对于所有的 \mathsf{ind}\in\mathsf{DB}(w)，计算 \mathsf{e}\gets\mathsf{Enc}(K_e, ind)，然后将 \mathsf{e} 添加到列表 \mathbf{t} 中
      • (\mathsf{TSet}, K_T)\gets \mathsf{TSetSetup}(\mathbf{T})
      • 输出密钥 (K_S, K_T) 以及加密的数据库 \mathsf{EDB} = \mathsf{TSet}

    查询协议：

    • 客户端已知密钥 (K_S, K_T)，以及查询的关键词 w
    • 计算 \mathsf{stag} \gets \mathsf{TSetGetTag}(K_T, w)，发送 \mathsf{stag} 给服务器
    • 服务器计算 \mathbf{t}\gets \mathsf{TSetRetrieve}(\mathsf{TSet}, \mathsf{stag})，返回 \mathbf{t} 给客户端
    • 客户端令 K_e\gets F(K_S, w)，对于在 \mathbf{t} 中的每一个 \mathsf{e}，计算 \mathsf{ind}\gets \mathsf{Dec}(K_e, \mathsf{e})，然后输出 \mathsf{ind}

    如何将其扩展为支持连接搜索 (conjunctive keyword search)？

    比如输入的查询是 \bar{w} = (w_1, \ldots, w_n)，对于在 \bar{w} 中的每个关键词 w_i，都执行一次。但是服务器并不直接将 \mathbf{t} 返回给客户端，而是从客户端出获得 K_{e_i}，其中 i=1,\ldots, n，然后对 \mathsf{e} 解密得到每个 w_i 的 \mathsf{ind} 集合。这样搜索的复杂度为 \sum_{i=1}^n |\mathsf{DB}(w_i)| 的倍数。

    3. Basic Cross-Tags 协议 （BXT）

    假设客户端给的关键词列表为 \bar{w} = (w_1, \ldots, w_n)，假设 \mathsf{DB}(w_1) 是最短的。首先运行 SKS 搜索协议来查找关键词 w_1，客户端得到所有匹配 w_1 的文件。

    BXT 协议的核心思想是使用 SKS 来检索 \mathsf{TSet}(w_1)，然后服务器对 w_2,\ldots,w_n 的结果取交集，服务器只会返回 full conjunction 的结果。

    具体的做法是在 \mathsf{EDBSetup}(\mathsf{DB}) 阶段，除了生成 \mathsf{TSet} 之外，还生成一个 \mathsf{XSet}，其中包含的元素我们称为 \mathsf{xtag}。对每个 w\in W，首先计算关键词的陷门 \mathsf{xtrap} = F(K_X, w)；对于每一个 \mathsf{ind}\in\mathsf{DB}(w)，需要计算 \mathsf{xtag} = f(\mathsf{xtrap}, \mathsf{ind})。

    对于连接搜索 (w_1,\ldots, w_n)，选择估测出现频率最低的关键词，比如 w_1，使用 SKS 协议，即 K_e\gets F(K_S, w_1)，\mathsf{stag}\gets \mathsf{TSetGetTag}(K_T, w_1)。对于每一个 i=2,\ldots, n，客户端计算 \mathsf{xtrap}_i\gets F(K_X, w_i)，发送 (K_e, \mathsf{stag}, \mathsf{xtrap}_2, \ldots, \mathsf{xtrap}_n)。服务器使用 \mathsf{stag} 检索 \mathbf{t} = \mathsf{TSetRetrive}(\mathsf{TSet}, \mathsf{stag})。对每一个在 \mathbf{t} 中的密文 \mathsf{e}，首先解密 \mathsf{ind} = \mathsf{Dec}(K_e, \mathsf{e})，如果 f(\mathsf{xtrap}_i, \mathsf{ind})\in \mathsf{XSet}，发送符合条件的 \mathsf{ind} 给客户端。

    这样一来，服务器就知道了 \mathsf{ind}

    BXT 协议的问题在于，当服务器知道了 \mathsf{xtrap}_i 之后，便可以进行保存。在之后的操作中，它便可以知道对于任意给定的 \mathsf{ind}，只需要测试 是否 f(\mathsf{xtrap_i},\mathsf{ind})\in \mathsf{XSet}，便可以知道该文件索引对应的文件中是否包含关键词 w_i。

    一种规避这种攻击的方法是，让客户端计算该函数而不是让服务器执行。但这样一来通信开销就会增大。另一种方式是使用一种 oblivious shared computation between client and server 来代替之前的函数 f(\mathsf{xtrap}, \cdot)。

    首先，我们考虑使用盲指数，将 \mathsf{ind} 盲化，即 \mathsf{xind} = F_p(K_I, \mathsf{ind})，那么 \mathsf{xtag} 将被定义为 g^{F_p(K_X,w)\cdot \mathsf{xind}}。客户端首先计算并发送 g^{F_p(K_X, w_i)\cdot z}，其中 z\in Z_p^* 为盲因子。服务器将其作为 \mathsf{xind} 的指数，服务器最后将收到的结果使用 z^{-1} 去盲，即得到 g^{F_p(K_X, w_i)\cdot \mathsf{xind}}。

    这种方式可靠吗？

    这种方法也有问题，服务器会知道 g^{F_p(K_X, w_i)}，同样也会采取跟之前类似的攻击。

    4. Oblivious Cross-Tags 协议 （OXT）

    在设计 OXT 协议之前，首先需要解决两个问题。核心思想是在 \mathsf{EDBSetup} 预计算盲的部分，并将其保存在 \mathsf{EDB} 中。也就是说，对于每个关键词以及对应的索引元组，首先保存一个盲值 y_c = \mathsf{xind}\cdot z_c^{-1}，其中 z_c 是由 w 和计数器 c 生成。

    计数器的目的是为了让每个 z 值都不一样

    搜索时，服务器需要对每一个 \mathsf{xind} 计算 g^{F_p(K_X, w_i)\cdot \mathsf{xind}}，并且在 \mathsf{XSet} 中检验它们是否存在。 对于在 \mathbf{t} 中的第 c 个元组，客户端定义长度为 n 的数组 \mathsf{xtoken}[c]，即 \mathsf{xtoken}[c,i]:=g^{F_p(K_X, w_i)\cdot z_c}，其中 i=1,\ldots, n。 z_c 是根据 w 和计数器 c 盲化后的值。服务器通过 T-set 找出 w_1 匹配的结果，然后通过测试 \mathsf{xtoken}[c,i]^{y_c}\in \mathsf{XSet} 是否成立来过滤第 c 个结果。由于 \mathsf{xtoken}[c,i]^{y_c} = g^{F_p(K_X,w_i)\cdot z_c \cdot \mathsf{xind}\cdot z_c^{-1}} = g^{F_p(K_X,w_i)\cdot \mathsf{xmind}}

    相对于 BXT 协议，OXT 泄露的信息更少。对于服务器而言，它既不知道 \mathsf{ind} 也不知道 \mathsf{xtrap_j}，j=2,\ldots, n。
